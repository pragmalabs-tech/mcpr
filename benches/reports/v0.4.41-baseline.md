# mcpr v0.4.41 — baseline reference bench

First reference run using the `benches/` harness. Pins a number we can compare
future changes against. **Includes one confirmed protocol bug** (SSE framing
not byte-passed) that blocks one scenario — captured in Findings below.

## Methodology

```
Hardware:   Apple M4, 10 cores, 16 GB RAM
OS:         macOS 25.3.0 (Darwin)
Rust:       rustc 1.92.0 (ded5c06cf 2025-12-08)
mcpr:       0.4.41
Load tool:  oha 1.14.0   (no coordinated-omission correction)
Mocks:      stateless (hand-rolled axum), stateful (rmcp 1.5.0)
Topology:   client, proxy, mock all on loopback, no CPU pinning
Warmup:     3 s
Run length: 8–10 s per scenario
Runs:       1 (reference snapshot — multi-run median is future work)
Date:       2026-04-19
```

**Caveats baked into these numbers**:
- Single-run, not median-of-N. Tails noisier than they should be.
- Loopback only — no network RTT. Real deployments add it.
- oha lacks coordinated-omission correction; p99/p99.9 understate queueing.
- Concurrency pinned at 20 because 50+ exhausts macOS ephemeral ports at loopback rates.

## Config

```toml
# benches/configs/bench.toml
```

### 2. Realistic overhead — sweep upstream latency, 20 conns, 8 s each

Same `tools/call echo` workload. As upstream gets slower, proxy overhead %
collapses toward zero — while the absolute µs delta stays roughly constant.

| Upstream | Direct p50 | Proxied p50 | Δ p50   | Direct rps | Proxied rps | Loss   |
|----------|-----------:|------------:|--------:|-----------:|------------:|-------:|
| 0 µs     | 103 µs     | 295 µs      | +192 µs | 170,019    | 47,537      | −72.0 %|
| 1 ms     | (noisy)*   | 4.74 ms     | —       | —          | 4,617       | —      |
| 10 ms    | 12.23 ms   | 12.29 ms    | +54 µs  | 1,659      | 1,602       | −3.4 % |
| 100 ms   | 102.27 ms  | 102.65 ms   | +378 µs | 197        | 195         | −1.3 % |

\* 1 ms tier had partial failures at 20 conns (baseline dropped to 62 % success
due to localhost port churn). Needs a re-run with lower concurrency.

**The takeaway**: mcpr's overhead is a **roughly fixed per-request cost**,
dominated by the pipeline (parse → middleware → forward → middleware → emit).
Against realistic upstreams (≥10 ms), the percentage impact is <5 % — in the
noise. The scary `−77 %` number from Scenario 1 is **the upstream being
unrealistically fast**, not mcpr being slow.

## Findings

### Finding 1 (critical): SSE framing is not byte-passed on POST /mcp responses

Upstream SSE:
```
content-type: text/event-stream
transfer-encoding: chunked

data:
id: 0
retry: 3000

data: {"jsonrpc":"2.0","id":1,"result":{...}}
```

Through mcpr:
```
content-type: text/event-stream
content-length: 149             ← buffered (not streamed)
access-control-allow-origin: *  ← proxy-added CORS

data: {"id":1,"jsonrpc":"2.0",...}   ← stripped id:/retry:/empty data:
                                     ← JSON keys reordered (re-serialized)
```

mcpr currently:
1. Buffers the SSE stream (`transfer-encoding: chunked` → `content-length: N`)
2. Strips non-`data:` SSE metadata (`id:`, `retry:`, empty prefix lines)
3. Deserializes and re-serializes the JSON payload (reorders keys)
4. Re-emits with proxy-generated CORS headers

Real MCP clients (rmcp included) fail to parse this. The unit-test suite
misses it because no test drives a streaming upstream through mcpr with a
spec-compliant client.

**Suspect code** (from targeted search in `crates/mcpr-core/src/proxy/`):
- `pipeline/run.rs:123` — unconditional `read_body_capped(resp, …)` buffers every POST response, including SSE streams.
- `sse.rs:10-14` — `extract_json_from_sse()` keeps only `data:` lines with content, discarding `id:`, `retry:`, and empty-prefix lines.
- `pipeline/middleware/sse.rs:138` — `serde_json::to_vec(...)` re-serializes, reordering keys.

**Fix direction**: detect `content-type: text/event-stream` on the upstream
response *before* buffering; return
`Body::from_stream(resp.bytes_stream())` and skip the SSE unwrap/wrap/rewrite
middleware chain entirely.

**Reproduce**: `benches/scripts/scenarios/sse-compat.sh` (returns exit 1).

### Finding 2 (investigate): p95+ tail blows out for proxied traffic

Across all three latency scenarios, mcpr's p95 was 3–10× worse than direct.
At 0 µs upstream, proxied p99 was **5.83 ms** vs **360 µs** direct — a
5.5 ms blowout on the tail even though p50 delta was only 191 µs.

Possible causes (not investigated in this run):
- Event-bus `try_send` backpressure when the 10k channel fills
- `RwLock<RewriteConfig>` contention under burst
- `MemorySessionStore` locking (for the stateful case)
- Allocator behavior on the `Arc<Bytes>` path

Worth a follow-up bench with the observability sinks disabled to rule in/out
the event bus as the primary tail contributor.

### Finding 3 (minor): `bench.sh` concurrency > 20 exhausts macOS ephemeral ports

At 50+ concurrent oha connections on loopback, the local port range (16384–
65535) fills with TIME_WAIT sockets within seconds and both baseline and
proxied fail with `EADDRNOTAVAIL`. Not a mcpr bug — an artifact of loopback
+ high rps + macOS defaults. Mitigated by pinning `CONNECTIONS=20` in the
scripts. Fuller matrix sweep would need either (a) a Linux runner with
higher `net.ipv4.ip_local_port_range`, or (b) a unix-socket mock variant.

## Interpretation — what to take away

- **Fixed proxy overhead for plain forwarding**: ~200 µs at p50, ~1 ms at p95.
  Against any realistic upstream this is <5 % impact.
- **Session-creation cost**: ~40 µs additional at p50 when initialize fires.
- **Tail latency is worse than expected** — flagged for investigation, don't
  publish as an SLO.
- **mcpr has a real SSE forwarding bug** that this harness surfaced and the
  existing unit tests missed. Fixing it also unblocks realistic
  session-reuse benchmarking.
- **Percentages in Scenario 1 (−77 %) are misleading** — upstream doing zero
  work makes mcpr's fixed overhead look like a huge fraction. Always report
  absolute µs deltas; let readers compute % against their own upstream.
