# mcpr v0.4.41 — local-build reference

Companion to `v0.4.41-baseline.md` (which benched the prebuilt release from
GitHub). This report re-runs the same scenarios against the working-tree
`cargo build --release` to isolate whether the release build has a
performance gap vs. locally-built code.

## Methodology

```
Hardware:   Apple M4, 10 cores, 16 GB RAM
OS:         macOS 25.3.0 (Darwin)
Rust:       rustc 1.92.0 (ded5c06cf 2025-12-08)   ← much newer than release CI
mcpr:       0.4.41 (working tree, one commit past v0.4.41 tag — harness only)
Load tool:  oha 1.14.0
Mocks:      stateless (hand-rolled axum), stateful (rmcp 1.5.0)
Topology:   all on loopback, no CPU pinning
Warmup:     3 s
Run length: 8–10 s per scenario
Runs:       1 per scenario (single run — tails noisy)
Date:       2026-04-19
```

**Local build provenance**:
- Built via `cargo build --release -p mcpr-proxy --bin mcpr`
- No `rust-toolchain.toml` in repo → uses host rustc (1.92.0, very recent stable).
- No `[profile.release]` override → cargo defaults (`opt-level=3`, `lto=false`, `codegen-units=16`, `panic=unwind`).
- No `RUSTFLAGS` set → generic aarch64-apple-darwin codegen.
- No `.cargo/config.toml` → no target overrides.

**Release-CI build provenance** (from `.github/workflows/release.yml`):
- `dtolnay/rust-toolchain@stable` at CI time → whichever stable was latest when v0.4.41 was cut.
- `cargo build --release --target aarch64-apple-darwin` (explicit target triple).
- Same `[profile.release]` defaults as local.
- Source is byte-identical (only one commit since tag, it's the bench harness).

**Revised conclusion**: with a second measurement, the installed vs local
gap is **modest — ~10-15 % throughput, median latencies effectively
identical, tails within run-to-run variance**. The dramatic 68 % /
84 % deltas reported in an earlier one-shot smoke test were mostly
noise. Single-run numbers on loopback are unreliable for tails —
multi-run medians are needed for a credible claim.

**What's real across both builds**:
- Per-request overhead at p50: **~200 µs**. Consistent.
- Throughput ceiling: **38k-42k rps** at 20 conns. Consistent.
- Tail p99 blowout: **5-8 ms regardless of build**. Consistent — this is a
  real mcpr characteristic, not a build artifact.

## Findings specific to this report

### Finding A (methodology): single-run tails are noisy

The gap between the "63 k rps, p99 921 µs" smoke run and this proper
10 s measurement (41 k rps, p99 5.76 ms) is too large to be real code
differences. Loopback bench runs benefit enormously from warm caches, low
TIME_WAIT pressure, and clean scheduler state — all of which change second
to second on a dev laptop.

**Action**: the harness should support a `--runs N` knob that reports
median-of-N with MAD (median absolute deviation). Until then, treat all
single-run tail numbers as ±30–50 %.

### Finding B (real, not build): tail latency is a genuine mcpr characteristic

Both builds show proxied p99 at 5–8 ms while p50 is at 200–600 µs.
That's a **~20× gap between median and tail** — far worse than a well-tuned
Rust async proxy should exhibit (usual p99/p50 ratio is 2–4×).

The tail is the real story, not the build. Same candidate causes as
baseline report (`rewrite.rs` deserialize+reserialize allocation storms,
event-bus dispatcher scheduler contention, reqwest pool churn). Still
worth running under `samply` to confirm.

## Release-build improvement opportunities

Independent of whether there's a measurable gap, here are changes that
would make the CI release artifact faster and more reproducible. Ranked by
cost/benefit:

### 1. Pin the Rust toolchain (free, high confidence)

Add `mcpr/rust-toolchain.toml`:
```toml
[toolchain]
channel = "1.92"   # or the latest stable you've tested
profile = "minimal"
components = ["rustfmt", "clippy"]
```
Removes "CI drifts with stable" as a variable. Also prevents surprise
regressions when a new rustc changes inlining or codegen.

### 2. Tune `[profile.release]` (3-5 min longer builds, ~10-20 % runtime win typical)

In workspace `Cargo.toml`:
```toml
[profile.release]
lto = "thin"          # cross-crate inlining, ~5–15 % on hot paths
codegen-units = 1     # better optimization, slower build
panic = "abort"       # smaller binary, small runtime win
strip = true          # size only, no perf impact — optional
```
`lto = "thin"` is almost always a win without the massive build-time cost
of `lto = "fat"`. `codegen-units = 1` completes the picture. These two
together commonly produce the biggest single-change throughput gains on
async Rust services.

### 3. Swap the default allocator (5 lines, ~5-10 % on alloc-heavy paths)

mcpr allocates per-request: `Bytes`, `HeaderMap`, deserialized JSON
structures, reqwest buffers. macOS's default allocator is competent but
not great. Add to `mcpr-cli/src/main.rs`:
```toml
# in mcpr-cli/Cargo.toml
[dependencies]
mimalloc = { version = "0.1", default-features = false }
```
```rust
// main.rs (top of file)
#[global_allocator]
static GLOBAL: mimalloc::MiMalloc = mimalloc::MiMalloc;
```
Trivially reversible if it regresses. `jemallocator` is an alternative
with similar gains on Linux but less compelling on macOS.

### 4. Eliminate the response-rewrite alloc on non-JSON bodies (code change, unknown impact)

The SSE bug and the p99 tail might share the same root cause: the
response-body path unconditionally reads into `Bytes`, optionally
deserializes, URL-rewrites, and reserializes. For responses that aren't
JSON-with-URLs (most tool-call payloads, all SSE streams), this is
wasted work. Fixing it would:
- Solve the SSE framing bug (the big win)
- Cut one allocation + one JSON traverse per request on the fast path
- Probably shrink the tail materially

This is the highest-impact code change. The SSE fix is the forcing function.

### 5. Skip where possible — things NOT worth doing now

- **`-C target-cpu=native`**: breaks distributed binaries. Not for release.
- **PGO / BOLT**: great gains (10-20 %) but heavy CI complexity. Revisit if items 1–4 leave a gap.
- **Replacing reqwest**: large surface-area change for uncertain win.

## Recommended action

1. **Immediately**: adopt items 1 and 2 (rust-toolchain + `[profile.release]`).
   These are one-commit changes, well-understood, reversible. Re-bench and see.
2. **Next**: fix the SSE forwarding bug (from baseline report). This may
   also shrink the p99 tail as a side effect.
3. **After that**: profile under `samply` to see if the tail is still
   meaningful. If yes, try item 3 (mimalloc). If still a problem,
   investigate item 4 specifically in the rewrite pipeline.
4. **Harness debt**: add multi-run support to the bench scripts before
   publishing any new numbers. Current results have ±30-50 % tail noise
   that's larger than most of the changes above will produce.
